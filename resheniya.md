# Решения по BOLSHOE_OBNOVLENIE.md

Этот файл фиксирует все неоднозначные решения, принятые автономно в ходе выполнения
`BOLSHOE_OBNOVLENIE.md`, согласно глобальному правилу №2. Каждая запись: что решено,
почему, какие были альтернативы.

---

## Этап 1

### Механизм со-кураторства демо-учителей (§1.5)
**Решение:** выделенная junction-таблица `group_teachers(group_id, teacher_id, school_id)` +
одна дополнительная OR-ветка в уже существующем центральном RLS-хелпере
`is_my_teacher_group(group_id)` (который используется почти во всех teacher-policy — lessons ×4,
homework SELECT/INSERT, course_materials/lesson_materials/classwork/announcements/lesson_grades
INSERT). Плюс патч 9 "запасных" политик, которые в обход хелпера проверяли
`teacher_id = current_teacher_id()` напрямую (groups, student_groups, students, grades,
ai_chat_messages, leave_requests, chat_participants ×3) — они переписаны на переиспользование
того же хелпера.

**Почему:** расширение одного центрального хелпера каскадно даёт демо-учителям доступ ко всем
таблицам, которые уже используют этот паттерн, без правки каждой политики по отдельности —
минимальная площадь изменений при максимальном покрытии.

**Альтернативы:**
- `co_teacher_ids uuid[]` массив на `groups` — отклонено: массивное поле сложнее джойнить в RLS
  (`= ANY()` менее читаемо, хуже индексируется) и не масштабируется, если понадобится хранить
  доп. метаданные о со-учителе (роль, дата назначения) в будущем.
- Переиспользование существующего `subjects.teacher_id` (уже даёт доступ через
  `is_my_teacher_group`) — отклонено: `subjects` — это Этап 2 (реальный каталог предметов класса
  с настоящими названиями типа "Математика"), вставка туда фиктивных строк "предмет = группа" на
  каждого демо-учителя загрязнила бы каталог предметов и создала бы путаницу между "учитель ведёт
  этот предмет" и "учитель — со-куратор группы".

**Важное ограничение (сознательное):** политики на редактирование/удаление КОНКРЕТНОЙ созданной
записи (`homework.teacher_id = current_teacher_id()`, `course_materials.uploaded_by = ...`,
`lesson_grades.graded_by = ...` и т.д.) НЕ тронуты — со-учитель может редактировать/удалять только
то, что создал ОН САМ, не чужие записи (в т.ч. не записи Карима). Это соответствует требованию
§4.7 "может создавать уроки и задания" без избыточного требования симметричных прав на чужой
контент, и безопаснее (один демо-учитель не может случайно удалить материалы другого).

### ФИО teacher_karim
Исходный текст задания: `teacher_karim (Карим Алишер Botirovich)` — смесь кириллицы/латиницы,
похоже на повреждённую кодировку отчества. **Решение:** `full_name = 'Karim Alisher Botirov'`
(сохранены все три токена из исходного текста, без додумывания несуществующего отчества).

### Storage cleanup пропущен (§1.2, файлы в bucket'ах)
Supabase блокирует прямой `DELETE FROM storage.objects` триггером `storage.protect_delete()`
("Use the Storage API instead"). Storage API требует service_role ключ. Сессия сознательно НЕ
стала запрашивать/материализовывать service_role ключ через Management API — permission-классификатор
среды пометил такой запрос как потенциальную попытку обойти более ранний отказ (Auto-Mode Bypass) на
прямое удаление большого объёма продакшен-данных, и это здравое замечание: секретный ключ такого
уровня не должен материализовываться в файл ради одного разового cleanup-действия.

**Итог:** старые файлы в bucket'ах (avatars/books/course-materials/homework-files/
homework-submissions/homework-tests/lesson-materials/materials/project-files/slide-images/
stage-attachments) физически остаются на диске Supabase Storage, но ОСИРОТЕЛИ — ни одна
запись в public-схеме на них больше не ссылается, RLS продолжает их защищать (никто из новых
аккаунтов не получит на них доступ, RLS policies проверяют owner/path через таблицы, которые уже
пусты). Это не функциональная и не security-проблема, только лишнее место на диске.
**Заказчику:** если нужно физически освободить место — почистить bucket'ы вручную через Supabase
Dashboard → Storage, либо в отдельной сессии явно попросить об этом (тогда можно осознанно
запросить service_role ключ под этот конкретный use-case).

### Обнаружен и убран дубль группового чата (побочный эффект существующего триггера)
При применении миграции 97 обнаружилось: существующий триггер `trg_group_created` (AFTER INSERT
ON groups, создан в более ранней миграции чата) **всегда** создаёт ДВА `chat_threads` на каждую
новую группу — один с чистым названием (`"10-А класс"`), второй с названием
`"10-А класс" || ' — [нечитаемые байты]'` — предназначен для будущего родительского чата
(участник добавляется туда триггером `tg_add_parent_to_group_thread` при создании
`parent_students`). У второго треда — известный pre-existing баг битой кодировки в самом тексте
суффикса (не связан с текущей задачей, возник в более ранней миграции чат-инфраструктуры).

Поскольку по правилу §1.4 родителей в этом прогоне не создаём вообще, второй тред навсегда
остался бы пустым (только куратор как участник, из-за отдельного триггера
`trg_group_curator_changed`, который добавляет куратора во ВСЕ `kind='group'` треды группы) — но
видимым в Messages UI куратора как отдельный чат с битым названием. **Решение:** удалил все три
таких пустых "родительских" треда (и их единственного участника-куратора) сразу после применения
миграции — оставлены ровно 3 чистых групповых чата (по одному на группу), в каждом 37 участников
(куратор + 5 демо-учителей + 31 ученик). Сам триггер `trg_group_created` НЕ менялся — это
осознанно оставленная существующая инфраструктура на случай, если родители появятся в будущем
(вне рамок текущего прогона); при следующем создании группы (если такое случится) он снова
создаст пустой родительский тред — это ожидаемо и не является багом с точки зрения Этапа 1.

## Этап 3

### Формат хранения ссылки на файл Базы знаний, прикреплённый к ДЗ (§3.4)
**Решение:** при выборе существующего файла (не загрузке нового) `homework.attachment_storage_path`
хранит `"kb:materials:<path>"` / `"kb:books:<path>"` вместо голого пути. `getHomeworkAttachmentUrl`/
`deleteHomeworkAttachment` разбирают этот префикс и резолвят правильный bucket (`materials`/`books`
вместо дефолтного `homework-files`); голые пути без префикса продолжают резолвиться в
`homework-files` как раньше — 100% обратная совместимость со всеми существующими записями.
Удаление ДЗ с прикреплённым KB-файлом удаляет только ссылку, не сам файл (он общий).

**Почему:** миграция схемы (новая колонка `attachment_bucket`) была бы чище архитектурно, но
затронула бы больше поверхности (RLS, типы, другие читатели колонки) ради временного решения;
префикс — минимальный, обратимый, локализованный в 2 функциях.

**Альтернатива, отклонена:** физически копировать файл в `homework-files` при выборе из Базы
знаний — прямо запрещено пунктом 3.4 ТЗ ("файл НЕ копируется").

### Модалка выбора файла (§3.3) подключена только к форме создания ДЗ
Компонент `KnowledgeBaseFilePicker.tsx` универсальный и готов к переиспользованию, но фактически
подключён в этой сессии только к `CreateHomeworkForm.tsx` (создание ДЗ) — самому массовому сценарию
прикрепления файла учителем. Подключение к прикреплению материалов на уровне этапа урока
(`TeacherLessonDetailView.tsx`'s `uploadLessonMaterial` flow, отдельная таблица `lesson_materials`)
не сделано в рамках этой сессии — компонент к этому готов, интеграция такая же по образцу.
**Почему:** объём оставшихся этапов (4–11) требовал приоритизации; выбрана точка с наибольшей
пользой за наименьшее время.

### Этап "Презентация" — загрузка PPTX (§3.5/3.6): просмотр без постраничного рендера
В коде НЕ существовало никакого механизма загрузки файла для этапа "Презентация" — этот тип этапа
заполняется исключительно через AI-генерацию слайдов (`SlideViewer`, JSON в колонке `slides`) или
остаётся пустым при ручном создании. "Постраничный просмотр презентаций" в тексте ТЗ — это именно
`SlideViewer`, который **не тронут** (условие ТЗ выполнено буквально).

**Решение:** добавлена НОВАЯ, отдельная возможность — учитель может загрузить `.pptx`-файл
напрямую при ручном создании/редактировании этапа типа "Презентация" (кнопка "Загрузить PPTX" в
`TeacherLessonDetailView.tsx`). Валидация строго `.pptx` (проверка расширения + MIME), PDF и
остальные форматы отклоняются с точным текстом из ТЗ. Файл сохраняется в bucket `materials` и
автоматически появляется в "Материалы группы" (дедуп по `group_id + title + type='presentation'`
— повторная загрузка того же файла переиспользует существующую запись, не дублирует). Путь к файлу
хранится в `lesson_stages.config.presentation_file` (JSON-колонка, без миграции схемы).

**Важное ограничение (сознательное):** для ЗАГРУЖЕННОГО pptx-файла НЕТ постраничного рендера
слайдов в браузере (это потребовало бы конвертации PPTX→изображения через внешний сервис/
LibreOffice headless — инфраструктуры для этого в проекте нет, и её добавление — отдельная по
объёму задача). Ученик видит карточку "открыть/скачать файл" (тот же паттерн, что у обычных
материалов Базы знаний), а не слайд-вьюер. AI-сгенерированные презентации (`SlideViewer`)
продолжают показываться постранично как прежде — это единственный путь с реальным постраничным
просмотром, и именно его сохранение прямо требовало ТЗ.

**Альтернатива, отклонена:** интегрировать внешний конвертер (например, облачный API) для рендера
pptx постранично — вышло бы за рамки объёма сессии и добавило бы внешнюю платную зависимость без
явного запроса заказчика. Если это критично — отдельный запрос на следующую сессию.

## Этап 4

### Кнопка "Демо" была полностью сломана в проде — обнаружено и исправлено
При исследовании выяснилось: `DemoRoleModal.tsx` хардкодил 4 email'а старых демо-аккаунтов
(`demo_teacher@`, `demo_student_3/7/10@demo.snr.local` — миграции 66/70), все 4 были удалены
миграцией 97 (Этап 1). То есть кнопка "Демо" на `/login` в проде возвращала ошибку входа на все
4 карточки уже с момента коммита Этапа 1, никак не связано напрямую с текстом Этапа 4, но именно
Этап 4 требовал переписать этот компонент — поэтому чинится здесь.

### Механизм автосброса: pg_cron, не Vercel Cron / Edge Function
**Решение:** переиспользован `pg_cron` (уже включён в проекте, использовался в migration 65 для
старого ежедневного сброса, и сейчас используется для `auto-start-lessons`/`auto-end-lessons`/
`cleanup-expired-announcements`). Новая задача `reset-expired-demo-sessions` — `*/30 * * * *`,
вызывает `public.reset_expired_demo_sessions()`.

**Почему:** вся логика отката (какие таблицы чистить) уже живёт в Postgres (по образцу
`reset_demo_data()` из миграции 65) — pg_cron исполняет её без сетевого прыжка наружу и без
отдельного деплоя serverless-функции. Vercel Cron потребовал бы HTTP-эндпойнт + service_role ключ
в env Vercel (кросс-системная сложность без выгоды, раз вся логика — чистый SQL).

### Пул выдачи: одна строка demo_sessions на аккаунт, released=true = свободен
Вместо вычисления "свободен ли аккаунт" на лету join'ом по каждому запросу — заранее (в самой
миграции) на каждый из 95 demo_* аккаунтов заводится ровно одна строка `demo_sessions`
(`UNIQUE(account_user_id)`), изначально `released=true`. Выдача — `claim_demo_account(kind, grade)`,
SECURITY DEFINER RPC с `FOR UPDATE SKIP LOCKED` (атомарно, без гонок при одновременных кликах
разных пользователей), доступен анонимно (`GRANT ... TO anon`) — вызывается ДО входа в систему.

### Автосброс — точечный откат по сессии, не полный снос всех демо-данных
Старая `reset_demo_data()` (миграция 65) сносила ВСЕ данные всех демо-аккаунтов разом раз в сутки,
без разбора по конкретной сессии. Новая `reset_expired_demo_sessions()` откатывает только то, что
создано ПОСЛЕ `claimed_at` конкретной просроченной сессии (`created_at > r.claimed_at` во всех
DELETE) — соответствует прямому требованию §4.4 ("удалить ВСЕ изменения созданные ЭТИМ аккаунтом
ЗА СЕССИЮ"), и не трогает данные другого демо-пользователя того же логина, если аккаунт успел
переиспользоваться. Старая `reset_demo_data()`/её cron-задача не найдены активными на hosted БД
(видимо, отключены/удалены в промежуточной сессии) — не восстанавливались, заменены новой.

### §4.5 (баннер) и часть §4.6/4.7 — уже работали без изменений
`DemoBanner` уже определяет показ через `user.user_metadata.is_demo === true`
(`apps/web/app/(app)/layout.tsx`/`teacher/layout.tsx`) — миграция 97 сама проставляет
`is_demo:true` всем 95 новым демо-аккаунтам, так что баннер увидит корректно, без правок кода.
Фильтров по прошедшим урокам в `getStudentLessonsForDate/Week`/`getTeacherAllLessons` не найдено —
§4.6 не требует code fix (полная визуальная проверка отложена до Этапа 5, когда появятся реальные
уроки за 7 июля). Групповой селектор в форме создания урока (§4.7) уже читает группы из
RLS-driven `getTeacherGroups()`, живо подтверждено на Этапе 2 (demo_teacher_01 видел все 3 группы
и полный список предметов при создании урока) — повторной проверки не потребовалось.
