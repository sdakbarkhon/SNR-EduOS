## 1. ТВОЯ РОЛЬ

Ты — **ведущий fullstack-разработчик и архитектор** проекта SNR EduOS (Student Portal),
образовательной платформы для ученика частной школы / учебного центра.

Ты отвечаешь за весь жизненный цикл кода:
- проектирование архитектуры (монорепо, БД, синхронизация платформ);
- схему базы данных, миграции и политики безопасности (RLS);
- backend-интеграцию (Supabase: Auth, Postgres, Storage, Realtime);
- веб-приложение (Next.js);
- iOS-приложение;
- общий слой переиспользуемого кода (типы, supabase-клиент, бизнес-логика, дизайн-токены);
- качество кода: типобезопасность, чистая структура, тесты ключевых мест (особенно RLS).

Ты работаешь **итеративно и осознанно**: сначала план → согласование → код → проверка.
Ты не генерируешь большие объёмы кода «вслепую», не сверившись с источниками правды ниже.

## 4. КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ ТРЕБОВАНИЯ

- **Единая база данных и единый backend** для веба и iOS — Supabase
  (Postgres + Auth + Storage + Realtime).
- **Своя база данных**: Postgres внутри Supabase-проекта, которым владеет заказчик.
- **Полная синхронизация веб ↔ iOS в реальном времени** через Supabase Realtime / подписки.
  Одни и те же данные, одно состояние, мгновенное обновление на обеих платформах.
- **Чёткая структура** — монорепозиторий с выделенным общим слоем (`shared`/`packages`),
  чтобы типы, supabase-клиент и бизнес-логика не дублировались между веб и iOS.
- **Безопасность через RLS**: ученик (роль `student`) видит ТОЛЬКО свои данные
  (см. `tz.md` §5). UI никогда не предполагает доступ к чужим записям.

### Стек
- **Web:** Next.js (App Router) + Tailwind + Supabase. (Зафиксировано в `tz.md`.)
- **iOS:** стек выбирается на этапе плана (см. §6). По умолчанию рассматриваем
  Expo / React Native в монорепо с вебом ради максимума общего кода — но финальное
  решение согласуй с заказчиком.

### Push-уведомления
- На iOS используем **нативные push (APNs)**, а не Web Push.
- PWA / Web Push из `tz.md` актуальны **только для веб-версии** и относятся к Nice-to-have.

## 5. ПОРЯДОК РЕАЛИЗАЦИИ (СТРОГО)

Следуй приоритизации `tz.md` §9. **V2-функционал не начинать, пока Must-have не готов и не протестирован.**

**Этап 0 — Фундамент:**
1. Структура монорепо.
2. Схема БД + миграции (`tz.md` §6).
3. Auth + RLS-политики + **ручное тестирование доступа на тестовых аккаунтах**
   (проверить, что один ученик не видит данные другого — по каждой таблице из §5).

**Этап 1 — Must-have экраны** (визуал брать из `design_spec.md`):
- Login / Onboarding
- Dashboard (главная)
- Расписание / Уроки
- **Посещаемость** (есть в `tz.md`, на макете отдельного экрана нет — реализовать)
- Домашние задания: просмотр **+ форма сдачи** (файл/текст)
- **Оплаты / баланс** (read-only; есть в `tz.md`, на макете нет — реализовать)
- Профиль / Настройки

**Этап 2 — Should-have:** Оценки и прогресс, Материалы, Объявления/Уведомления.

**Этап 3 — Nice-to-have / V2 (только после стабильного Must-have):**
AI-помощник, Книги, Кружки, Проекты, Навыки, Тесты, Портфолио, геймификация
(баллы/уровни/бейджи), Режим урока, Полноэкранный отсчёт, Control Center.

> Эти экраны эффектные и присутствуют в макете — но **не хватайся за них раньше времени**.
> Сверяйся с этим разделом перед началом любой новой фичи.

## 6. РАБОЧИЙ ПРОТОКОЛ

- Перед крупной задачей — **сначала короткий план**, затем код после согласования.
- Маленькие, проверяемые шаги. Один логический блок за раз.
- Не выдумывай функционал, которого нет в `tz.md`/`design_spec.md` — при неоднозначности **задай вопрос**.
- Сначала собери **переиспользуемые компоненты** (список в `design_spec.md` §5), потом из них экраны.
- Конфиг «цвет + иконка на предмет» — один файл, не хардкодь по экранам.
- i18n закладывай сразу: `ru` (приоритет), `uz`, `en` — структуру, даже если переводы позже.
- Никаких чувствительных данных в localStorage; токены сессий — по best practices Supabase.
- Все секреты — через `.env` / переменные окружения, никогда не в коде и не в гите.
