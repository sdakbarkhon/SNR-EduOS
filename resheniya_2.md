### Переключение проектов — реализация

Чистый клиентский React state (`activeProjectId: string | null` в `CodeSandbox`), НЕ URL-параметр и НЕ отдельный роут. `null` = "Новый проект" (работает с автосейв-слотом текущего языка); установленный id = именованный проект (автосохранение делает `UPDATE` этой строки, а не создаёт новый автосейв). Переключение языка (`useState<Lang>`) сбрасывает `activeProjectId` в `null` и грузит автосейв нового языка — реализует "свой отдельный автосейв на переключение сервиса/языка" буквально.

### RLS — учитель НЕ видит чужие sandbox_projects (текст политик)

```sql
CREATE POLICY "sandbox_projects_select_own" ON public.sandbox_projects FOR SELECT
  USING (student_id = public.current_student_id());
-- + аналогично INSERT/UPDATE/DELETE
```
`current_student_id()` возвращает `NULL` для любой не-ученической сессии (teachers/admins/parents не имеют строки в `students`), а `student_id = NULL` в Postgres никогда не `TRUE` — учитель физически не может получить ни одной строки через эту политику, независимо от того, чья это запись. Никакого отдельного "учителям запрещено" правила не требуется — у таблицы просто нет ни одной политики, доступной не-владельцу.

### Расхождение с исходными данными пользователя

Пользователь перечислил 11 "реальных учеников" (sherzod_10, nodira_07, aziz_03, aziza_10, farrukh_10, diyora_10, bekzod_07, malika_07, sardor_07, rustam_03, zarina_03). Это не блокировало работу — автосоздание чатов реализовано data-driven (по факту student_groups/subjects/groups), а не по хардкоду списка юзернеймов, так что расхождение не потребовало ничего менять в подходе, только пересчитало ожидаемое количество тредов (18, не 66).

### Часть 1 — схема: расширение chat_threads, а не новая таблица

Проверка существующей схемы (миграции 78/79/81/82/83/84/97) показала: `chat_threads.kind` УЖЕ допускает `'direct'` (CHECK-constraint с самой миграции 78) — путь существовал, но ни разу не использовался в продакшене (миграция 78's легаси-перенос старых `public.messages`-строк нашёл 0 строк на момент миграции). Вся инфраструктура вокруг чата (RLS через `is_my_thread()`, realtime publication, query-слой `getMyThreadSummaries`/`getThreadMessages`/`sendChatMessage`/`markThreadRead`, компонент `MessagesView.tsx`) уже полностью generic по `kind` — реализация нового типа треда через существующие таблицы не требует НИЧЕГО нового в этих слоях, только точечных правок.

**Выбор**: не отдельная таблица `direct_chats`, а `chat_threads.student_id`/`chat_threads.teacher_id` — две nullable-колонки, заполняемые только при `kind='direct'`, по прямой аналогии с тем, как `group_id`/`title` уже являются "payload"-колонками для `kind='group'` (chat_threads — уже дискриминированный union по `kind`, эта модель просто продолжает существующий паттерн). Альтернатива "искать пару через self-join chat_participants" отклонена — она не даёт дешёвой uniqueness-гарантии (нужен был бы exclusion constraint через агрегат) и не даёт дешёвого RLS/enrichment-джойна из query-слоя без пары дополнительных self-join'ов на каждый запрос списка тредов.

**"Куратор" — не отдельный тип чата.** Он тот же personal-чат с учителем, который просто совпал с `groups.teacher_id` — UI сравнивает `chat_threads.teacher_id` с курирующим `teacher_id` группы ученика (см. `isCuratorThread` в query-слое) и подсвечивает эту строку. Дополнительная `role_in_thread='curator'` для direct-участника НЕ добавлялась — этот enum-статус остаётся зарезервирован за групповыми тредами, как и раньше.

### Часть 1 — область автосоздания: только реальные ученики

90 демо-учеников из пула УЖЕ сидят в `student_groups` (не появляются туда заново при клейме — клейм лишь помечает `demo_sessions`/сессию, не вставляет новую строку `student_groups`). Если бы автосоздание было data-driven без фильтра — 93 ученика × 6 учителей = 558 пустых тредов, из которых 540 — мусор для аккаунтов, в которые почти никогда не логинятся. chat_threads/chat_participants/chat_messages НЕ входят в список 10 is_demo-флагованных таблиц миграции 110 — значит обычная демо-сессия-чистка (`reset_demo_data_for_user`) их не подчищает, и этот мусор оставался бы навсегда. Решение: фильтр `username NOT LIKE 'demo\_%' ESCAPE '\'` (тот же паттерн, что миграция 110 использовала для удаления демо-учителей) — и в одноразовом backfill, и во всех трёх триггерах (student_groups INSERT, subjects.teacher_id INSERT/UPDATE, groups.teacher_id UPDATE). Итог на hosted: ровно 18 тредов (3 ученика × 6 учителей), 36 строк chat_participants — подтверждено прямым SELECT после применения.

Учителя-демо (demoLogin) не нуждались в отдельной обработке: демо-сессия учителя логинится ПОД РЕАЛЬНЫМ teacher-аккаунтом (teacher_math и т.д., is_demo_session — флаг сессии, не отдельная строка teachers) — значит демо-учитель автоматически видит те же personal-чаты, что и реальный, без всякой дополнительной RLS-логики.

### Часть 1 — RLS: единственная правка — исключить kind='direct' из admin-обхода

Существующие read/write-политики уже корректно ограничивают доступ через `is_my_thread()` (участие в chat_participants) — это уже гарантирует "ученик видит только свои чаты" и "учитель видит только чаты, где он участник" БЕЗ дополнительной логики: раз только настоящая пара (студент, учитель) когда-либо становится участником direct-треда (через `fn_ensure_direct_chat`), доступ третьих лиц структурно невозможен. Родители никогда не добавляются участниками direct-тредов — значит для требования "родители — доступа нет" вообще не нужно было ничего менять.

Единственная реальная дыра: у `fn_is_admin()` (школьный админ, НЕ super_admin) был блэнкет-байпас на чтение/редактирование ЛЮБОГО треда школы в 5 политиках (`chat_threads` SELECT/UPDATE/DELETE, `chat_participants` SELECT, `chat_messages` SELECT/UPDATE) — не завязанный на участие. Добавлено условие `kind <> 'direct'` (или эквивалентный `NOT EXISTS(... kind='direct')` там, где джойн к chat_threads нужен) в каждую из этих 5 политик. `is_super_admin()` остался сквозным бэкдором без изменений — по прецеденту ВСЕХ остальных RLS-политик проекта (announcements, lessons и т.д.), где super_admin — платформенный аварийный доступ, отдельный от школьного `fn_is_admin()`, к которому явно относилась инструкция "админ — доступа нет".

### Часть 2 — N+1 в getTeacherAnnouncements

Причина: последовательный `for`-цикл с одним `await`-запросом `announcement_reads` на каждую строку объявления (не даже `Promise.all`, а строго sequential). Фикс: один `.in("announcement_id", allIds)` + `Map<string, number>`-подсчёт — идиома, уже использованная в этом же файле (`gSize` через embedded-count) и в `getStudentProjects()` (packages/core/src/queries/projects.ts) для той же формы задачи "count детей на каждую родительскую строку". ### Часть 5 — Gemini: план сорван дневной квотой, вызванной собственным багом

Первый прогон скрипта упал на ВСЕХ 21 треде с ошибкой `null value in column "school_id"` — `chat_messages.school_id DEFAULT current_school_id()`, а `current_school_id()` резолвится через `auth.uid()`, которого при service-role вызове НЕТ (NULL). ### Часть 3: двухшаговый workaround для graded_by при проверке ДЗ

### Ключевое архитектурное решение: не трогать lessons.status вообще

Это одновременно (а) буквально следует инструкции пользователя не лезть в статус самостоятельно, (б) не противоречит требованию "не трогать уроки" — `lessons` как таблица не изменяется этим промтом вообще, только читается.

### SQL для ручного применения (миграция `127_demo_stage_activation_exempt.sql`, файл уже в репо, НЕ применена к hosted)

```sql
BEGIN;

CREATE OR REPLACE FUNCTION public.fn_stamp_is_demo()
 RETURNS trigger
 LANGUAGE plpgsql
AS $function$
BEGIN
  IF TG_OP = 'INSERT' THEN
    IF auth.uid() IS NOT NULL THEN
      NEW.is_demo := public.is_demo_session();
    END IF;
    RETURN NEW;
  END IF;

  IF TG_OP = 'UPDATE' AND TG_TABLE_NAME = 'lessons'
     AND to_jsonb(NEW) - 'active_stage_id' = to_jsonb(OLD) - 'active_stage_id' THEN
    RETURN NEW;
  END IF;

  IF TG_OP = 'UPDATE' AND TG_TABLE_NAME = 'lesson_stages'
     AND to_jsonb(NEW) - 'current_slide_index' = to_jsonb(OLD) - 'current_slide_index' THEN
    RETURN NEW;
  END IF;

  IF pg_trigger_depth() <= 1
     AND OLD.is_demo = false
     AND public.is_demo_session() THEN
    RAISE EXCEPTION 'editing_real_data_in_demo' USING ERRCODE = 'P0002';
  END IF;

  IF TG_OP = 'UPDATE' THEN
    NEW.is_demo := OLD.is_demo;
    RETURN NEW;
  END IF;

  RETURN OLD;
END;
$function$;

INSERT INTO supabase_migrations.schema_migrations (version)
VALUES ('127') ON CONFLICT (version) DO NOTHING;

COMMIT;
```

Почему `CREATE OR REPLACE` одной функции, а не новая политика/триггер: тот же единственный триггер
`trg_stamp_is_demo` продолжает существовать без изменений (`DROP`/`CREATE TRIGGER` не нужны) — меняется
только его функция. `to_jsonb(NEW) - 'col' = to_jsonb(OLD) - 'col'` означает "изменилась только эта
колонка, всё остальное в строке идентично" — если когда-нибудь код обновит `active_stage_id` вместе с
чем-то ещё реальным одним `UPDATE`, исключение НЕ сработает и блок вернётся (безопасно к будущим
изменениям, не только к текущему `setActiveStage`/`setCurrentSlide`, которые обновляют ровно одну
колонку).

**P2 учтено**: когда демо-режим станет просто "баннер поверх реального аккаунта" без отдельного пула, весь
`fn_stamp_is_demo()` (и `is_demo`-колонки под ним) скорее всего уйдёт целиком — тогда это исключение
удалится вместе с остальной функцией одним махом, а не потребует отдельного отката. Фикс нужен СЕЙЧАС, а
не может ждать P2: демо-режим прямо сейчас нерабочий для самой демонстрируемой фичи (живой урок), P2 не
имеет обозначенного срока в этой сессии.

## mobile-parent: "Ошибка соединения" у всех кроме менеджера (2026-07-15)

**Корень бага одной строкой**: гипотеза про build-time env vars НЕ подтвердилась — они уже были поправлены в
этой же сессии (`app.json`'s `expo.extra`, коммит `a1374e2`) и корректно доставлены OTA-апдейтом на Android

в фоне на холодном старте, но **применяет** его только на СЛЕДУЮЩЕМ холодном старте; явного

день, полностью закрывая/открывая его много раз — и рано или поздно "перепрыгнул" на исправленный бандл.

сломанном бандле неопределённо долго.

**Косвенное подтверждение из самого текста ошибки**: пользователи видят именно "Проверьте соединение"
(`d.parentMobile.networkError`), а НЕ "Ошибка конфигурации приложения..." (`d.parentMobile.configError`,
добавлен в Round 3 этой же сессии). Если бы у них воспроизводилась ТЕКУЩАЯ версия бага (`missing env vars`),
они бы видели ВТОРОЕ сообщение — они видят ПЕРВОЕ, generic-сообщение, которое существовало ДО того, как в
этой сессии вообще завели различение config/network ошибок. Это ещё один independent сигнал, что их

## P1-5-фикс: LocaleContext — переключатель языка работает по-настоящему (2026-07-15)

По итогам аудита выше заказчик явно попросил почитить обнаруженный баг (переключатель существовал, но
не распространялся на другие экраны и не переживал перезапуск) отдельным коммитом, с точным техническим
планом. Реализовано ровно по плану заказчика:

- **`apps/mobile-parent/src/i18n/LocaleContext.tsx`** (новый файл) — `LocaleProvider` (один `useState`
  на всё дерево, единственный источник правды) + `useAppLocale()` через `useContext`. Публичный
  интерфейс `{ locale, d, setLocale }` не изменился — это и обеспечило «минимальный diff»: ни один из
  ~30 экранов, вызывающих `useAppLocale()` из `"../i18n"`, править не пришлось вообще (0 правок, не
  просто «минимум»).
- **`apps/mobile-parent/src/i18n/index.ts`** — теперь просто `export { LocaleProvider, useAppLocale }
  from "./LocaleContext"` (старая реализация с голым `useState` удалена).
- **`apps/mobile-parent/App.tsx`** — `<LocaleProvider>` обёрнут вокруг `<SafeAreaProvider>` в ready-ветке
  рендера (там же, где раньше жили только `ErrorBoundary`/`SafeAreaProvider`/`RootNavigator`); LoginScreen
  тоже монтируется внутри Provider (он — часть `RootNavigator`), поэтому смена языка до входа сохраняется
  тем же механизмом, что и после.
- **Персистентность** — через `mockStorage.ts` (тот же expo-secure-store враппер, что уже несёт
  настройки уведомлений/PIN из МОБ-6), ключ **`mob6.language`**. При маунте `LocaleProvider` сначала
  берёт системную локаль устройства (`Localization.getLocales()[0]?.languageCode`, тот же уже рабочий
  паттерн, что был в старом хуке — не переключались на `Localization.locale`, такого поля в
  используемой версии `expo-localization` может не быть), затем асинхронно перечитывает сохранённый
  выбор из `mockStorage` и переопределяет стейт, если он есть.

**Одно осознанное отклонение от буквального плана заказчика** — «синхронизация с i18next»: в проекте

большим и незапрошенным архитектурным изменением, идущим вразрез с уже задокументированным решением —
поэтому «синхронизация с i18next» реализована как синхронизация с уже существующей системой словарей

### ⚠️ Живая находка: у GEMINI_API_KEY нет квоты на gemini-2.5-pro

Проверено напрямую (`node`-скрипт с реальным ключом, ЧАСТЬ 8): `gemini-2.5-flash` работает штатно
(текст, `responseMimeType: "application/json"`, `responseSchema`, `systemInstruction` + история — всё
живьём подтверждено). `gemini-2.5-pro` отвечает `429` с `"limit: 0"` — это **не** временное исчерпание
квоты (был бы ненулевой limit), а структурное отсутствие Pro-доступа на текущем бесплатном тарифе ключа.

`generate-stages` и `curriculum-plans/parse` (обе явно просились на Pro в ТЗ) остались на Pro по
умолчанию, но `gemini-client.ts` теперь **детектирует именно этот `"limit: 0"` паттерн** и автоматически
откатывается на Flash в рамках той же попытки (без обычного backoff-ретрая, который здесь бесполезен —
задержка ничего не изменит) — фичи продолжают работать (чуть более дешёвым качеством) вместо жёсткого
падения. При апгрейде `GEMINI_API_KEY` на платный тариф это условие просто перестанет срабатывать без
единой правки кода — Pro включится сам.

### Применённые оптимизации (ЧАСТЬ 5)

1. **Структурированный JSON-output** — `apps/web/lib/ai/schemas.ts`: строгая `responseSchema` для
   `generate-homework` (file/test/programming) и `curriculum-plans/parse`. `generate-stages` и bundle
   осознанно оставлены без схемы — полиморфная форма (slides разного layout, config зависит от типа
   подзадачи) не выражается responseSchema без риска потерять валидные варианты. Живьём проверено —
   ответ приходит чистым JSON без markdown-обёртки.
2. **Context caching** — задокументировано в `prompts.ts`: `chat()` уже передаёт системный промпт
   отдельным полем `systemInstruction` (правильная структура для implicit caching Gemini 2.5), но честно
   отмечено, что текущий `EDUOS_ASSISTANT_LESSON_CHAT_SYSTEM_PROMPT` (~150 токенов) НЕ достигает порога
   кэширования (~1024/2048 токенов) — реального кэш-хита пока не будет, задел на будущее расширение
   промпта.
3. **Обрезка истории чата** — `AiAssistantView.tsx`: раньше на каждый запрос уходила ВСЯ история чата
   (неограниченно растущий client-side массив); теперь — последние 15 сообщений (упрощённый вариант без
   summary, явно разрешённый ТЗ).
4. **Сокращение промптов** — все 5 системных/пользовательских промптов в `prompts.ts` переписаны в
   императиве без вежливых оборотов; у трёх ДЗ-промптов со строгой схемой убран теперь избыточный блок
   "ВЕРНИ СТРОГО JSON {...}" (schema уже гарантирует форму). `generate-stages`-промпт (самый большой, с
   тщательно настроенными по комментариям в коде эвристиками ключевых слов) тронут МИНИМАЛЬНО — только
   вступительная фраза — осознанное решение не рисковать регрессией в промпте, доведённом итерациями до
   рабочего состояния, без возможности живой проверки в этой сессии. JSON-контракты (имена полей, формат
   ответа) не менялись нигде.
