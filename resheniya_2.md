### Переключение проектов — реализация

Чистый клиентский React state (`activeProjectId: string | null` в `CodeSandbox`), НЕ URL-параметр и НЕ отдельный роут. `null` = "Новый проект" (работает с автосейв-слотом текущего языка); установленный id = именованный проект (автосохранение делает `UPDATE` этой строки, а не создаёт новый автосейв). Переключение языка (`useState<Lang>`) сбрасывает `activeProjectId` в `null` и грузит автосейв нового языка — реализует "свой отдельный автосейв на переключение сервиса/языка" буквально.

### RLS — учитель НЕ видит чужие sandbox_projects (текст политик)

```sql
CREATE POLICY "sandbox_projects_select_own" ON public.sandbox_projects FOR SELECT
  USING (student_id = public.current_student_id());
-- + аналогично INSERT/UPDATE/DELETE
```
`current_student_id()` возвращает `NULL` для любой не-ученической сессии (teachers/admins/parents не имеют строки в `students`), а `student_id = NULL` в Postgres никогда не `TRUE` — учитель физически не может получить ни одной строки через эту политику, независимо от того, чья это запись. Никакого отдельного "учителям запрещено" правила не требуется — у таблицы просто нет ни одной политики, доступной не-владельцу.

### Расхождение с исходными данными пользователя

Пользователь перечислил 11 "реальных учеников" (sherzod_10, nodira_07, aziz_03, aziza_10, farrukh_10, diyora_10, bekzod_07, malika_07, sardor_07, rustam_03, zarina_03). Это не блокировало работу — автосоздание чатов реализовано data-driven (по факту student_groups/subjects/groups), а не по хардкоду списка юзернеймов, так что расхождение не потребовало ничего менять в подходе, только пересчитало ожидаемое количество тредов (18, не 66).

### Часть 1 — схема: расширение chat_threads, а не новая таблица

Проверка существующей схемы (миграции 78/79/81/82/83/84/97) показала: `chat_threads.kind` УЖЕ допускает `'direct'` (CHECK-constraint с самой миграции 78) — путь существовал, но ни разу не использовался в продакшене (миграция 78's легаси-перенос старых `public.messages`-строк нашёл 0 строк на момент миграции). Вся инфраструктура вокруг чата (RLS через `is_my_thread()`, realtime publication, query-слой `getMyThreadSummaries`/`getThreadMessages`/`sendChatMessage`/`markThreadRead`, компонент `MessagesView.tsx`) уже полностью generic по `kind` — реализация нового типа треда через существующие таблицы не требует НИЧЕГО нового в этих слоях, только точечных правок.

**Выбор**: не отдельная таблица `direct_chats`, а `chat_threads.student_id`/`chat_threads.teacher_id` — две nullable-колонки, заполняемые только при `kind='direct'`, по прямой аналогии с тем, как `group_id`/`title` уже являются "payload"-колонками для `kind='group'` (chat_threads — уже дискриминированный union по `kind`, эта модель просто продолжает существующий паттерн). Альтернатива "искать пару через self-join chat_participants" отклонена — она не даёт дешёвой uniqueness-гарантии (нужен был бы exclusion constraint через агрегат) и не даёт дешёвого RLS/enrichment-джойна из query-слоя без пары дополнительных self-join'ов на каждый запрос списка тредов.

**"Куратор" — не отдельный тип чата.** Он тот же personal-чат с учителем, который просто совпал с `groups.teacher_id` — UI сравнивает `chat_threads.teacher_id` с курирующим `teacher_id` группы ученика (см. `isCuratorThread` в query-слое) и подсвечивает эту строку. Дополнительная `role_in_thread='curator'` для direct-участника НЕ добавлялась — этот enum-статус остаётся зарезервирован за групповыми тредами, как и раньше.

### Часть 1 — область автосоздания: только реальные ученики

90 демо-учеников из пула УЖЕ сидят в `student_groups` (не появляются туда заново при клейме — клейм лишь помечает `demo_sessions`/сессию, не вставляет новую строку `student_groups`). Если бы автосоздание было data-driven без фильтра — 93 ученика × 6 учителей = 558 пустых тредов, из которых 540 — мусор для аккаунтов, в которые почти никогда не логинятся. chat_threads/chat_participants/chat_messages НЕ входят в список 10 is_demo-флагованных таблиц миграции 110 — значит обычная демо-сессия-чистка (`reset_demo_data_for_user`) их не подчищает, и этот мусор оставался бы навсегда. Решение: фильтр `username NOT LIKE 'demo\_%' ESCAPE '\'` (тот же паттерн, что миграция 110 использовала для удаления демо-учителей) — и в одноразовом backfill, и во всех трёх триггерах (student_groups INSERT, subjects.teacher_id INSERT/UPDATE, groups.teacher_id UPDATE). Итог на hosted: ровно 18 тредов (3 ученика × 6 учителей), 36 строк chat_participants — подтверждено прямым SELECT после применения.

Учителя-демо (demoLogin) не нуждались в отдельной обработке: демо-сессия учителя логинится ПОД РЕАЛЬНЫМ teacher-аккаунтом (teacher_math и т.д., is_demo_session — флаг сессии, не отдельная строка teachers) — значит демо-учитель автоматически видит те же personal-чаты, что и реальный, без всякой дополнительной RLS-логики.

### Часть 1 — RLS: единственная правка — исключить kind='direct' из admin-обхода

Существующие read/write-политики уже корректно ограничивают доступ через `is_my_thread()` (участие в chat_participants) — это уже гарантирует "ученик видит только свои чаты" и "учитель видит только чаты, где он участник" БЕЗ дополнительной логики: раз только настоящая пара (студент, учитель) когда-либо становится участником direct-треда (через `fn_ensure_direct_chat`), доступ третьих лиц структурно невозможен. Родители никогда не добавляются участниками direct-тредов — значит для требования "родители — доступа нет" вообще не нужно было ничего менять.

Единственная реальная дыра: у `fn_is_admin()` (школьный админ, НЕ super_admin) был блэнкет-байпас на чтение/редактирование ЛЮБОГО треда школы в 5 политиках (`chat_threads` SELECT/UPDATE/DELETE, `chat_participants` SELECT, `chat_messages` SELECT/UPDATE) — не завязанный на участие. Добавлено условие `kind <> 'direct'` (или эквивалентный `NOT EXISTS(... kind='direct')` там, где джойн к chat_threads нужен) в каждую из этих 5 политик. `is_super_admin()` остался сквозным бэкдором без изменений — по прецеденту ВСЕХ остальных RLS-политик проекта (announcements, lessons и т.д.), где super_admin — платформенный аварийный доступ, отдельный от школьного `fn_is_admin()`, к которому явно относилась инструкция "админ — доступа нет".

### Часть 2 — N+1 в getTeacherAnnouncements

Причина: последовательный `for`-цикл с одним `await`-запросом `announcement_reads` на каждую строку объявления (не даже `Promise.all`, а строго sequential). Фикс: один `.in("announcement_id", allIds)` + `Map<string, number>`-подсчёт — идиома, уже использованная в этом же файле (`gSize` через embedded-count) и в `getStudentProjects()` (packages/core/src/queries/projects.ts) для той же формы задачи "count детей на каждую родительскую строку". ### Часть 5 — Gemini: план сорван дневной квотой, вызванной собственным багом

Первый прогон скрипта упал на ВСЕХ 21 треде с ошибкой `null value in column "school_id"` — `chat_messages.school_id DEFAULT current_school_id()`, а `current_school_id()` резолвится через `auth.uid()`, которого при service-role вызове НЕТ (NULL). ### Часть 3: двухшаговый workaround для graded_by при проверке ДЗ

### Ключевое архитектурное решение: не трогать lessons.status вообще

Это одновременно (а) буквально следует инструкции пользователя не лезть в статус самостоятельно, (б) не противоречит требованию "не трогать уроки" — `lessons` как таблица не изменяется этим промтом вообще, только читается.

- `backfill-chat-messages.mjs` / `backfill-announcements.mjs`: `runDailyTrickle`/`runDailyCadence` останавливаются на дне "сегодня" (+05:00), для самого "сегодня" время сообщения/объявления капается на `now()`.
- `run-daily-backfill.mjs`: логика не менялась — гейт целиком внутри под-скриптов, поэтому `toDate` теперь можно смело передавать далеко вперёд (хоть до конца расписания) без риска будущих данных.

## Промт 7.6 — автопубликация материалов урока в БЗ при завершении урока (2026-07-13)

### Аудит (Часть 1) — реальная схема разошлась с планом в промте

- `lesson_materials`: id, lesson_id, title, `file_storage_path` (не `storage_path`), file_size_bytes, file_original_name, uploaded_by, created_at, `visibility` (`'all'|'teacher_only'`), school_id, is_demo, from_knowledge_base, kb_bucket. **Нет колонки file_type** — MIME выводится по расширению в самой функции.
- `course_materials`: id, group_id, **lesson_id** (уже есть! FK → lessons ON DELETE SET NULL, миграция 119), title, type, file_url, link_url, created_at, description, **subject** (свободный text, не subject_id FK), file_type, storage_path, file_size_bytes, uploaded_by, school_id, is_demo, stage_id. Отдельная колонка `source_lesson_id` из плана промта — **не нужна**, переиспользовали `lesson_id`.
- **КРИТИЧНАЯ находка, не было в плане промта**: `lesson_materials` физически хранит файлы в Storage-бакете `"lesson-materials"`, но ВЕСЬ существующий код `course_materials` (в частности `getMaterialDownloadUrl`) жёстко резолвит бакет `"materials"`. Прямая вставка `storage_path` без учёта бакета сломала бы скачивание автопубликованных материалов (404 — файл ищется не в том бакете). Решение: добавили `course_materials.bucket` (`text NOT NULL DEFAULT 'materials'`, все существующие 112 строк не затронуты), `getMaterialDownloadUrl` получил опциональный 4-й параметр `bucket`, `materials.ts`'s `getMaterialUrl` и `deleteMaterial` его читают/пробрасывают. Заодно поймали связанный риск: `deleteMaterial` раньше безусловно удалял файл из Storage при удалении строки — для `bucket='lesson-materials'` строк это стёрло бы файл, которым всё ещё владеет исходная `lesson_materials`-запись внутри урока; добавили пропуск удаления файла для этого случая.
- `lesson_materials.visibility='teacher_only'` — не публикуем в общую БЗ (следует из модели видимости, отдельно не оговаривалось в промте).
- `fn_auto_end_lessons` (миграция 85): `SECURITY DEFINER`, тело — обычный `UPDATE ... SET status='completed'` без `session_replication_role`/`pg_trigger_depth`-обхода → **триггеры срабатывают штатно**, наш `AFTER UPDATE OF status` реагирует одинаково что от cron-автозавершения, что от ручного клика "Закончить урок".
- RLS INSERT на `course_materials` требует `is_my_teacher_group(group_id) AND uploaded_by=current_teacher_id()` — не мешает, наша функция `SECURITY DEFINER` и обходит RLS целиком (запускается от владельца функции, а не от вызывающей сессии).
- Побочная находка (не в скоупе 7.6, задокументирована и не тронута): фильтр по предмету в `MaterialsView.tsx`/`TeacherMaterialsView.tsx` использует `m.group.subject` — устаревшее единое поле группы (`groups.subject='programming'` у всех 3 групп, см. баг `d76b687` из более ранней сессии), а не собственный `course_materials.subject` каждой строки. Видимость материалов это не блокирует (базовая выборка скоупится только по `group_id`), ломает только сужение по предмету в выпадающем списке.

### Миграция 124

`fn_lesson_materials_to_kb(p_lesson_id)` (SECURITY DEFINER) + `fn_lesson_status_to_kb()` + `AFTER UPDATE OF status ON lessons` триггер `trg_lesson_completed_to_kb`. Фильтр публикации: `from_knowledge_base=false AND visibility='all' AND file_storage_path IS NOT NULL`. Дедуп — `NOT EXISTS` по `(group_id, storage_path)`. `subject` заполняется из `subjects.name` урока. Ретроспективно НЕ применяется — только будущие переходы в `completed`.

### Тестовый прогон (Часть 3, на синтетическом тестовом уроке 10-А/Программирование с датой 2099 года)

5 SELECT'ов:
1. `course_materials` до завершения — 0 строк.
2. `course_materials` после первого `UPDATE status='completed'` — 3 новые строки, `bucket='lesson-materials'`, `subject='Программирование'`, `file_type` верно по расширению (pdf/docx/png), `lesson_id`=тестовый урок.
3. Дедуп-проверка (COUNT) — сначала буквальный повторный `UPDATE status='completed'` (не меняет `OLD.status`, триггер даже не вызывает функцию — защита на уровне `IF NEW.status='completed' AND OLD.status IS DISTINCT FROM 'completed'`), затем — более строгий тест: переключили статус на `in_progress` и обратно на `completed`, чтобы триггер РЕАЛЬНО вызвал функцию второй раз. Результат: `count(*)=3, count(DISTINCT storage_path)=3` — сработал `NOT EXISTS`-дедуп внутри самой функции, не только внешний guard триггера.
4-5. `DELETE FROM lesson_materials` → `DELETE FROM lessons` (тестовый урок) — `course_materials` сохранил все 3 строки, `lesson_id` корректно упал в `NULL` (`ON DELETE SET NULL`), сам тестовый урок и его `lesson_materials` полностью исчезли (cascade). После подтверждения пользователя 3 тестовые строки `course_materials` удалены отдельным DELETE (тест оставлял их в реальной группе 10-А со ссылками на никогда не загруженные Storage-пути — не нужно оставлять в проде).
