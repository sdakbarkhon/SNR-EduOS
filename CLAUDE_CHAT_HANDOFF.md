> **Как использовать:** скопируй весь этот файл целиком в первое сообщение нового чата (планировщика/Claude Code).
> Это полный актуальный контекст проекта на 2026-07-22: что сделано, что применено к БД, что осталось, что дальше.

## 1. О ПРОЕКТЕ

**SNR EduOS — образовательная платформа частной школы/учебного центра** (Ташкент, UTC+5). Основной язык интерфейса — русский, i18n на ru/uz/en заполняется сразу для новых строк во всех трёх файлах.

**Стек и структура (монорепо pnpm + Turborepo):**
- `apps/web` — Student Portal, Next.js 15 (App Router) + Tailwind + Supabase. Роли: ученик, учитель, родитель (частично), админ/суперадмин. Деплой на Vercel.
- `apps/mobile-parent` — приложение для **родителей** (не учеников), Expo/React Native. iOS — через Expo Go (OTA `eas update` на канал `preview`, реальной `.ipa` пока нет — нужен Apple Developer аккаунт, ~$99/год, заказчик пока не оформил). Android — полноценный APK через `eas build`.
- `apps/mobile` — ученическое приложение. **ЗАМОРОЖЕНО НАВСЕГДА**, не развивать без явного запроса.
- `apps/h5p` — не в скоупе, не трогать без явного запроса.
- `packages/core` — общий TS-слой: типы, Supabase-клиент, query-функции, i18n-словари (`src/i18n/{ru,en,uz,types}.ts`).
- `supabase/migrations` — SQL-миграции, **последняя применённая — 146**, следующий свободный номер — **147**. Миграции создаются как файлы в репозитории, но применяются к hosted-БД **вручную заказчиком** через Supabase Dashboard SQL Editor (Claude Code не имеет и не должен иметь прямого доступа на запись к hosted Postgres в обычном режиме работы).

**Роли и доступ:** `student`, `teacher`, `parent` (частично), `admin`/`superadmin`. Изоляция данных — через RLS-политики (Postgres Row Level Security), не через клиентскую логику.

## 2. GIT-ИНФРАСТРУКТУРА (важно, прочитать перед первым push)

- Код передан заказчику: репозиторий **`sdakbarkhon/SNR-EduOS`**.
- `origin` настроен на **dual-push**: обычный `git push origin main` уходит **в оба** репозитория одновременно — `MIPTKBoy/SNREduOS` (рабочий) и `sdakbarkhon/SNR-EduOS` (заказчика). `git fetch`/`git pull` — только из `MIPTKBoy`. Vercel деплоит именно с `MIPTKBoy`.
- **Важно:** всё, что попадает в `main`, сразу видно заказчику — включая промежуточные технические коммиты, не только финальные. При коммитах — осмысленные сообщения, без внутренней «кухни».
- **Инцидент безопасности (закрыт):** Supabase Personal Access Token (`sbp_...`) был обнаружен GitHub push protection в истории `resheniya_2.md` (коммит `745f177`). Токен отозван, локальные транскрипты Claude Code с ним удалены. **Правило на будущее:** реальные токены/секреты никогда не пишутся в `.md`-файлы репозитория — только имена переменных окружения (`SUPABASE_SERVICE_ROLE_KEY`, и т.п.), сами значения — только в `.env.local` (не в git).
- `VERCEL_TOKEN` лежит в `apps/web/.env.local`. Для запросов к Vercel API **обязательно** указывать `teamId` (аккаунт `miptkboy`, `team_MYpBV2M3L64gwy59SGSL6QCG`) — без него API молча возвращает пустой список деплоев, что легко принять за «токен недоступен».

### Миграции 143-146 (применены вручную через Dashboard, все живут в проде)

| 143 | Сняты с расписания pg_cron джобы `auto-start-lessons`/`auto-end-lessons` (функции НЕ удалены, только сняты с cron — есть закомментированный SQL для возврата) | Заказчик потребовал ручное управление уроками, никакой авто-логики по времени |
| 144 | `SECURITY DEFINER` + `search_path` на `fn_enqueue_lesson_stage_embedding`/`fn_enqueue_quiz_stage_embedding` | Фикс бага: миграция 139 создала эти триггеры как `SECURITY INVOKER`, очередь embedding закрыта `REVOKE ALL FROM authenticated` — любая вставка этапа падала 403 |

| 146 | RLS-политика `"student starts own scheduled lesson"` (`scheduled→in_progress`) | Симметрична миграции 117 (`in_progress→completed`) — ученик получил полный ручной контроль над своим уроком |

### Что готово и стабильно работает

1. **Ручное управление уроками** (не авто) — и учитель, и ученик могут начать/закончить урок отдельными кнопками, использующими одни и те же `startLesson()`/`endLesson()`. «Выйти из урока» у ученика — отдельное, не завершающее урок действие. Статус «Пропущен» убран целиком (его не было в БД, только в UI); прошедший-непроведённый урок показывается как «Запланирован». Метка «Сейчас» замирает на последнем уроке дня, не пропадает раньше полуночи. Крон `close-past-lessons` (19:00 UTC = 00:00 Ташкент) закрывает только вчерашние-и-старше уроки.
2. **30 учеников** (по 10 на группу 10-А/7-А/3-А), сокращено с 96 — критерий отбора: полнота данных (оценки/посещаемость/ДЗ/чаты), не происхождение username. Демо-вход теперь всегда попадает на одного из этих 30 полных аккаунтов.
3. **Презентация у ученика без блокирующего оверлея** — full-width слайд с локальным скроллом, отдельный «режим фокуса» (не браузерный Fullscreen API) вместо принудительной полноэкранной модалки. Realtime-синхронизация слайдов (`SlideViewer.tsx`, канал `stage-slide-*`) не тронута.
4. **AI-ассистент ученика**: LaTeX-формулы (KaTeX) и графики функций (`function-plot` + безопасный `mathjs`-парсер, не `eval`) прямо в чате.
5. **Страница «Оценки»**: фильтр Все/За задания/За урок + модалка деталей (что сдано, комментарий) — **только просмотр**, правка оценки прямо в модалке пока не сделана (техдолг).
6. **Чат — Telegram-раскладка**: три независимые зоны скролла, умный автоскролл. Правка была не в самом чате, а в трёх ролевых шеллах (`AppShell`/`TeacherShell`/`ParentShell`), у всех был безусловный `overflow-y-auto` на `<main>`.
7. **Учебные планы**: карточка плана кликабельна → страница деталей с правкой тем + 3 способа создания уроков из темы (авто с 1 августа / вручную по одному / заготовка в форму). Идемпотентность — через `lessons.curriculum_topic_id`.

## 4. ИЗВЕСТНЫЕ ОТКРЫТЫЕ ЗАДАЧИ ПО ВЕБУ (требования утверждены заказчиком, реализация впереди)

### Задания (`/homework`) — раздел не доработан

1. Починить нестыковку «0 из N подзадач выполнено» при уже выставленной оценке (bundle-задания) — счётчик должен считаться по реальным сдачам подзадач, а не замирать на 0.
2. Ученик внутри задания видит свою сдачу: код/текст/файл/ответы теста, для bundle — по каждой подзадаче отдельно; явный статус («отправлено на проверку»/«оценено»/«не сдано»); оценку; комментарий учителя.
3. Учитель видит полную сдачу ученика **и может** поставить оценку + написать комментарий прямо там, включая перезапись уже выставленной (это отличает раздел «Задания» от страницы «Оценки», где сейчас только просмотр).
4. Скоуп — **все** активные предметы (Программирование, Робототехника, Математика, Английский, Русский), ничего не скрывать и не удалять.
5. Источник сдач — **`homework_submissions`** (не `assessments` — такой таблицы в проекте нет вообще).
6. Может понадобиться RLS-миграция на `UPDATE homework_submissions` учителем — проверить `pg_policies` перед тем, как писать новую; если нужна — следующий свободный номер **147**.

### Учебный план 2.0 — расширение полей темы

- Каждая тема плана получает поля: длительность урока, цель урока, процесс (ход) урока, результат урока — плюс опционально доп. материалы и ссылки.
- AI предзаполняет эти поля при парсинге загруженного PDF/DOCX плана; учитель может править вручную.
- При создании урока из темы все эти поля переносятся в сам урок (цель/процесс/результат/материалы/ссылки/длительность).
- Этапы урока (теория/тест) из плана **не создаются** автоматически — отдельная кнопка AI-генерации этапов, которую учитель нажимает сам после создания урока.

## 5. ТЕХНИЧЕСКИЙ ДОЛГ (не блокирует демо заказчику, держать в уме)

- **Ротация `SUPABASE_SERVICE_ROLE_KEY`.** Проект на legacy-ключах (`eyJ...`) — ротация по одному невозможна, только roll JWT-секрета целиком (разлогинивает всех, ломает прод до обновления ключей в Vercel). Чистый путь — переезд на новый формат `sb_secret_...` (уже доступен в Dashboard), но требует проверки всех серверных роутов.
- **Оптимизация старта урока.** `router.refresh()` при каждом обновлении страницы урока заново генерирует signed URL для всех материалов. Упирается в осознанную security-границу (`PreLessonView` намеренно не видит `lesson_stages` до старта, чтобы не спойлерить контент) — нужен либо кэш signed URL, либо пересмотр границы.
- **Правка оценок учителем в модалке «Оценок»** — сейчас там только просмотр.
- **`students.curator_id`** («персональный куратор») не покрыт автосозданием direct-чатов — сейчас 0 расхождений с `groups.teacher_id`, эффекта нет, задел на будущее.
- **Вложения `project_submissions` в модалке «Оценок»** — в БД пока нет данных с реальными вложениями, путь чтения не построен (не на чём проверить).

### Мобилка — разблокировано

- Google-аккаунт `snreduos@gmail.com` **разблокирован 2026-07-22** (апелляция удовлетворена). Expo/EAS-доступ восстановлен, OTA-публикации снова возможны — **все мобильные задачи сняты с паузы**.
- **Напоминание при публикации OTA:** публиковать для **обеих** платформ сразу — не указывать `--platform`, либо явно `--platform all`. Android APK на канале `preview` подхватывает обновление автоматически при следующем запуске приложения.
- `apps/mobile` (ученическое) — **заморожено навсегда**, не трогать без явного запроса. В скоупе мобилки — только `apps/mobile-parent` (родительское приложение, iOS через Expo Go + Android APK).

### Desktop — новое направление, требования от заказчика

- Технология: **Tauri**.
- Платформа: **только Windows**, .exe. macOS/Linux — вне скоупа.
- Роли: **учитель и ученик**. `parent`/`admin` — отклоняются при попытке входа (десктоп не для них).
- Название приложения: **«SNR EduOS»**, версия **0.0.1**.
- Иконка — взять из `apps/mobile-parent`.
- Автообновление — через статический `.exe` на прод-домене + JSON-манифест (без сторонней update-инфраструктуры).
- **Без code signing** (сертификат не оформлен, осознанное решение заказчика на старте).
